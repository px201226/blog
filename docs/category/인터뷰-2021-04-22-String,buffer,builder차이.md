> **객체 지향 프로그래밍(OOP)이 뭐에요?**

객체 지향 프로그래밍은 컴퓨터 프로그래밍 패러다임중 하나로, `프로그래밍에서 필요한 데이터를 추상화시켜 상태와 행위를 가진 객체를 만들고 그 객체들 간의 유기적인 상호작용을 통해 로직을 구성하는 프로그래밍 방법`이다.


> **객체 지향 프로그래밍의 장, 단점 간단하게 설명해주세요**

### **장점**

- **코드 재사용이 용이**

남이 만든 클래스를 가져와서 이용할 수 있고 상속을 통해 확장해서 사용할 수 있음.

- **유지보수가 쉬움**

절차 지향 프로그래밍에서는 코드를 수정해야할 때 일일이 찾아 수정해야하는 반면 객체 지향 프로그래밍에서는 수정해야 할 부분이 클래스 내부에 멤버 변수혹은 메서드로 있기 때문에 해당 부분만 수정하면 됨.

<!-- more -->

- **대형 프로젝트에 적합**

클래스단위로 모듈화시켜서 개발할 수 있으므로 대형 프로젝트처럼 여러명, 여러회사에서 개발이 필요할 시 업무 분담하기 쉽다.

### **단점**

- **처리속도가 상대적으로 느림**
- **객체가 많으면 용량이 커질 수 있음**
- **설계시 많은 시간과 노력이 필요**

> **객체 지향 프로그래밍 키워드 5가지**

1. **정보은닉**
2. **추상화**
3. **캡슐화**
4. **상속**
5. **다형성**

> **클래스와 인스턴스(객체)는 무엇인지 설명해주세요.**

- **클래스** : 객체의 공통된 속성과 행동을 정의한 명세서
- **인스턴스(객체)** : 클래스를 통해 생성된 구체적인 실체

> **객체지향프로그래밍에서 추상화 (자료의 추상화)**

- 요구사항으로부터 필요한 부분과 불필요한 부분을 식별하고, 공통적인 부분을 정의하는 것

> **정보은닉이란 무엇인가요?**

- 객체지향 언어적 요소를 활용하여 객체에 대한 구체적인 정보를 노출시키지 않도록 하는 기법
- 기능의 교체나 변경에 대한 유연성을 제공
- 캡슐 내의 요소들에 대한 세부 구현사항들을 외부로부터 숨김
- 정보은닉의 개념이 캡슐화 개념안에 포함되기 하지만, 캡슐화가 되어있다고 해서 반드시 정보은닉이 되는 것이 아니다.

> **캡슐화가 무엇인가요?**

- 데이터와 알고리즘을 하나로 묶는 것  → 코드의 `재사용`이 용이해진다.
- 데이터를 공용 메서드롤 통해서만 접근을 허용한다.
- 관련된 요소들을 묶음으로써 캡슐 내부와 외부를 구별지어 특정 객체 속에 있는 데이터와 함수들을 다른 객체 속에 있는 데이터와 함수들과 구별이 이뤄짐

> **상속은 무엇인가요?**

- 부모 클래스의 속성과 기능을 물려 받아, 재정의와 확장을 통해 속성과 기능을 물려 받는 것으로,  기존의 코드를 `재활용` 할 수 있다.
- 객체들간의 계층적인 관계를 형성한다.
- 캡슐화의 한계를 보완한다. → 캡슐화를 통해 코드의 `재사용성`이 용이해졌지만, 시스템의 요구사항이 미세하게 달라도 재사용의 한계가 있었다. 상속을 이용하면 기존 코드를 `확장` 하여 필요한 부분을 재정의하거나 추가하여 재활용할 수 있다.  ex) 서블릿을 상속하거나 안드로이드의 액티비티를 상속하여 입맞에 맞게 커스텀한다. Spring Security의 UserDetail 클래스도 마찬가지다.

> **다형성은 무엇인가요?**

- 한 객체가 여러 개의 타입을 가질 수 있는 것을 말한다. (상속, 인터페이스)
- 한 타입의 참조변수로 여러 타입의 객체를 참조할 수 있도록 함
- 실제 동작은 다르더라도 개념적으로 동일한 작업을 하는 메서드를 같은 이름으로 사용할 수 있다.

**오버라이딩** : 부모클래스의 메서드와 같은 이름, 매개변수를 재정의 하는것.

**오버로딩** : 같은 이름의 함수를 여러개 정의하고, 매개변수의 타입과 개수를 다르게 하여 매개변수에 따라 다르게 호출할 수 있게 하는 것.

> **getter, setter 를 사용하는 이유는 무엇인가요?**

- getter, setter를 통해 간접접근하기 때문에, 매개변수로 넘어오는 값을 검증하거나 제한할 수 있다.
- 데이터 접근 의도를 추상화하여 메서드 이름으로 표현할 수 있다.
- 하지만 IDE 나 롬복에서 기계적으로 생성하는 getter, setter 는 크게 의미가 없다.
- 왜냐하면, 데이터 접근 의도가 추상화 되어 있지 않기 때문에 직접 접근과 차이가 없기 때문이다.
- 그래서 DTO(Data Transfer Object)의 경우에는 별도의 접근에 대한 추상화가 필요하지 않기 때문에 필드를 public으로 하기도 한다.
