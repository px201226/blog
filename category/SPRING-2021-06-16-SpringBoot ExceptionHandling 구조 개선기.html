Spring boot Exception Handling 구조 개선기
2021-06-16 > Spring/SpringBoot

# 배경

Spring boot 사용 시 Spring 에서 제공하는 @ControllerAdvice 과 @ExceptionHandler 어노테이션을 사용하면 예외 처리를 쉽게 모듈화 시킬 수 있습니다. @ExceptionHandler에 처리하고 싶은 예외를 입력하여 예외를 처리할 수 있을 것 입니다.

@ControllerAdvice
public class MyRestControllerAdvice { 

      ...

		  @ExceptionHandler(InvalidRequestParamaterException.class) 
			public ResponseEntity handle(InvalidRequestParamaterException e) {
					String message = "잘못된 입력입니다";
					return new ResponseEntity(message, HttpStatus.INTERNAL_SERVER_ERROR);
			}

	    ...
 }

저의 프로젝트도 위와 같은 방식으로 예외를 처리하고 있었습니다. 그런데 이와 같은 방식에 예외 처리 구조는 몇가지 문제점이 있습니다.

  • 처리해야 될 예외가 많아 질 수록 Advice 클래스가 비대해집니다.
  • Exception과 HTTP 상태 코드가 분리되어 Advice 클래스에서 이를 맵핑하고 있습니다. 때문에 예외와 매칭되는 HTTP 상태 코드를 한 눈에 파악하기도 힘들고, 예외와 HTTP 상태코드가 밀접한 관련이 있음에도 불구하고 응집도가 떨어집니다.
  • 예외를 구별하기 위해 이름만 다른 예외를 처리하기 위해서 매번 비슷한 핸들링 코드를 작성해야 합니다.

이와 같은 문제를 해결하기 위해 예외처리 구조를 개선해보겠습니다.

# 리팩토링

# 1. 예외 코드를 관리하는 Enum 클래스 생성

import static java.net.HttpURLConnection.*;

@Getter
@JsonFormat(shape = JsonFormat.Shape.OBJECT)
public enum  ErrorType {

    NO_SUCH_ITEM          (-101, HTTP_NOT_FOUND,      "해당 아이템이 존재하지 않습니다."),
    OUT_OF_STOCK_QUANTITY (-102, HTTP_INTERNAL_ERROR, "재고가 부족합니다."),
    ARGUMENT_NOT_VALID    (-103, HTTP_BAD_REQUEST,    "입력값이 잘못되었습니다.");

    @JsonIgnore
    private int httpStatus;
    private int errorCode;
    private String message;

    ErrorType(int errorCode, int httpStatus, String message ) {
        this.errorCode = errorCode;
        this.httpStatus = httpStatus;
        this.message = message;
    }

}

ErrorType 이라는 Enum 클래스를 만들고, 에러코드, HTTP 상태 코드, 간단한 메시지를 가지는 Enum 타입을 만듭니다. 이 때, 중요한 것은 HTTP 상태 코드를 java.net.HttpURLConnection 패키지 에서 불러옵니다. 스프링 프레임워크에 org.springframework.http 패키지에도 HTTP 상태코드가 정의되어 있지만, 스프링 프레임워크에 의존성이 생기므로 표준 자바에 정의되어 있는 HTTP 상태 코드를 사용합니다. 이렇게 하면 나중에 스프링 프레임워크에 의존성이 없으므로 공통모듈로 분리하기도 용이해집니다.

BusinessException 이라는 추상클래스를 만들고 1번에서 만든 ErrorType을 반환하는 추상 메서드를 정의합니다. 그리고, 이를 구현하는 NoSuchItemException, OutOfStockQuantityException 클래스를 만들고 getErrorType 추상메서드를 오버라이딩 하여 예외에 맞는 에러 코드를 반환하도록 구현합니다.

팩터리 패턴을 통해 Exception에서 ErrorType 인스턴스를 반환하도록 하여, 예외와 에러코드, HTTP 상태코드를 한 묶음으로 다룰 수 있도록 만들어 주었습니다.

사실 여기까지만 해도 충분히 예외 핸들링을 하는데 문제는 없지만, 에러 ResponseEntity를 만드는 클래스를 정의합니다.

# 3. 에러 응답 ResponseEntity 만들기

@Getter
@Builder
@JsonInclude(JsonInclude.Include.NON_NULL)
@NoArgsConstructor(access = AccessLevel.PROTECTED)
@AllArgsConstructor(access = AccessLevel.PROTECTED)
public class ErrorMessage {

    private ErrorType errorType;
    private String dispatcherPath;
    private String details;

    public static ResponseEntity of(ErrorType errorType, WebRequest request, String details){
        ErrorMessage messageBody = ErrorMessage.builder()
                .errorType(errorType)
                .dispatcherPath(((ServletWebRequest) request).getRequest().getRequestURI())
                .details(details)
                .build();

        return new ResponseEntity(
                messageBody,
                HttpStatus.valueOf(errorType.getHttpStatus())
        );
    }

}

# 4. ControllerAdvice 클래스 수정

@RestControllerAdvice
public class OrderControllerAdvice {

    @ExceptionHandler(BusinessException.class)
    public ResponseEntity handleBusinessException(BusinessException e, WebRequest request){
        return ErrorMessage.of(e.getErrorType(), request, e.getMessage());
    }

    @ExceptionHandler(MethodArgumentNotValidException.class)
    public ResponseEntity handleMethodArgumentNotValidException(MethodArgumentNotValidException e, WebRequest request){

        Map<String, String> fieldErrorMap = getFieldErrorMap(e);

        return ErrorMessage.of(ErrorType.ARGUMENT_NOT_VALID, request, fieldErrorMap.toString());
    }

    private Map<String,String> getFieldErrorMap(MethodArgumentNotValidException e){

        Map<String,String> fieldErrorMap = new HashMap<>();

        BindingResult bindingResult = e.getBindingResult();
        for (FieldError fieldError : bindingResult.getFieldErrors()) {
            fieldErrorMap.put(fieldError.getField(),fieldError.getDefaultMessage());
        }

        return fieldErrorMap;
    }

}

ControllerAdvice 에서 예외를 핸들링 할 때, BusinessException을 상속받는 모든 서브클래스를 하나의 메서드로 처리할 수 있도록 하였습니다.

그리고, BusinessException 이 아닌 유효성 검증 예외인 MethodArgumentNotValidException 와 같은 예외는 따로 정의하여 처리하도록 하였습니다.

개선전과 비교했을 때, Advice 클래스가 예외만 핸들링하고, 메세지 생성이라던지 HTTP 상태코드를 지정하는 등 같은 기능들은 다른 클래스에 위임하도록 하여 객체지향적으로 구조를 개선할 수 있었습니다.
